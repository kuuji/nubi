# PR Pipeline Summary Comment — Design

## Goal

When the monitor creates a PR, it also posts a structured comment summarizing the entire pipeline run. The PR description stays as a narrative (what changed and why), while the comment provides the detailed stage-by-stage breakdown — machine-readable data formatted for humans.

## Why a comment, not the PR body?

- The PR body is the monitor's narrative summary — it's written for reviewers who want to understand the change
- The pipeline summary is operational metadata — it's for people who want to understand *how* the change was produced
- Keeping them separate means the PR body stays clean and the pipeline data can be updated on retries without rewriting the description

## What it looks like

```markdown

**Task:** `add-rate-limiting` · **Branch:** `nubi/add-rate-limiting-a1b2c3`

### Executor
| | |
|---|---|
| Status | ✅ Complete |
| Attempts | 2 |
| Commit | `a1b2c3d` |
| Summary | Added rate limiting middleware with token bucket algorithm |

### Gates
| Gate | Result | Details |
|---|---|---|
| ruff | ✅ pass | 0 errors |
| radon | ✅ pass | max complexity: B (6) |
| pytest | ✅ pass | 12 passed, 0 failed |
| diff_size | ✅ pass | +87 -4 lines |

<details>
<summary>Gate details (attempt 1 — failed)</summary>

| Gate | Result | Details |
|---|---|---|
| ruff | ❌ fail | 3 errors (unused import, line too long, ...) |
| pytest | ⏭ skipped | blocked by lint failure |

</details>

### Reviewer
| | |
|---|---|
| Decision | ✅ Approve |
| Feedback | Clean implementation. Token bucket is the right choice for this use case. |

### Monitor
| | |
|---|---|
| Decision | ✅ Approve |

---
*Generated by [Nubi](https://github.com/kuuji/nubi) · [pipeline artifacts](.nubi/add-rate-limiting-a1b2c3/)*
```

## Data sources

All the data already exists — no new collection needed:

| Section | Source file | Already written by |
|---|---|---|
| Executor | `.nubi/{task_id}/result.json` | Executor agent |
| Gates | `.nubi/{task_id}/gates.json` | Gate runner (inside executor) |
| Reviewer | `.nubi/{task_id}/review.json` | Reviewer agent |
| Monitor | `.nubi/{task_id}/monitor.json` | Monitor agent |

The monitor already reads all of these via the GitHub API to produce its narrative summary. The pipeline comment is just a different rendering of the same data.

## Implementation

### Where the code goes

`src/nubi/tools/github_api.py` — add a `post_pipeline_summary` function alongside the existing `submit_audit` tool.

### When it runs

Inside the monitor agent's flow, after `submit_audit` creates/updates the PR. The monitor calls `post_pipeline_summary(pr_number)` as a follow-up step.

### What it does

1. Read all four artifact files from the branch (already has this code)
2. Format into the markdown template above
3. Post as a PR comment via `POST /repos/{owner}/{repo}/issues/{pr_number}/comments`
4. On subsequent runs (retries), update the existing comment instead of posting a new one (find by a marker like `<!-- nubi-pipeline-summary -->`)

### Scope

- ~50-80 lines for the formatter (JSON → markdown tables)
- ~10 lines for the GitHub API call (similar to existing PR creation code)
- Template handles missing data gracefully (e.g. if review was skipped, show "skipped")
- Collapsible `<details>` sections for failed gate attempts (can have multiple)

## Good candidate for a Nubi task

This is a self-contained feature with:
- Clear input (the four JSON files) and output (a markdown comment)
- Existing code patterns to follow (`submit_audit` in `github_api.py`)
- Testable — mock the JSON, verify the markdown output
- No architectural changes — it's additive to the monitor's existing flow

A TaskSpec for this would look like:

```yaml
spec:
  description: |
    Add a post_pipeline_summary function to the monitor that posts a structured
    PR comment after creating a PR. The comment should render the pipeline run
    data (executor result, gate results, reviewer decision, monitor decision)
    as formatted markdown tables. Read the existing submit_audit tool in
    src/nubi/tools/github_api.py for patterns. Update the monitor agent's
    system prompt to call this tool after submitting the audit.
  type: code-change
  inputs:
    repo: kuuji/nubi
    branch: main
    files_of_interest:
      - src/nubi/tools/github_api.py
      - src/nubi/agents/monitor.py
      - src/nubi/monitor_entrypoint.py
  review:
    enabled: true
    focus: [correctness, test_coverage]
```
